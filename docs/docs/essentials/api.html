API ​

API ​

/src/apps 是一个"魔法"目录，其下的所有的代码，只要具有名为 api 的导出，就将自动加载为路由。你可以根据需要在这里创建任意的目录结构，也请随意地在这里编写代码。

/src/apps is a "magic" directory where all the code under it will be automatically loaded as routes if it has an export named api. You can create any directory structure you need here and feel free to write code.

试试简单地向世界问好…？ (๑´ㅂ`๑)

Simply say hello to world..? (๑´ㅂ`๑)

ts
/**
 * 这是一个向世界问候的API！
 * 这些 ~~注释~~ 会被呈现到 **Cookbook** 当中
 */
export const api = defineApi({
  meta: {},
  async action(
    params: string & typia.tags.MinLength<2> & typia.tags.MaxLength<16>,
    context
  ) {
    return `你好世界! (来自 ${params})`
  }
})
ts
/**
 * This is an API that greets world!
 * These ~~comments~~ will be presented by the **Cookbook**
 */
export const api = defineApi({
  meta: {},
  async action(
    params: string & typia.tags.MinLength<2> & typia.tags.MaxLength<16>,
    context
  ) {
    return `hello world! (by ${params})`
  }
})

可以看到，定义 API 的方式和编写一个普通的方法很像！action 就是 API 所执行的具体动作，也就是你的逻辑。你可以返回任何内容，无论是字符串、对象还是数组，最终它们会被自动序列化为 JSON（准确地说，是 TSON）。其中，导出的名称 api 是固定的，一个文件中，只可以导出一个 API。

You can see that defining an API is very similar to writing a regular method! The action represents the specific action that the API performs, which is essentially your logic. You can return any content, whether it's a string, object, or array, and they will be automatically serialized into JSON (to be precise, TSON). The exported name api is fixed, and only one API can be exported in a file.

你可能会好奇，什么是 meta？第一个参数 params 和第二个参数 context 的用处都是什么？这些概念，在下面都会为你解答。

You may be curious about what meta is and what are the purposes of the first parameter params and the second parameter context. These concepts will be explained below.

Params ​

Params ​

params 来决定你的 API 可以接受哪些值。神奇的是，Milkio 会自动使用 Typia，来帮助确保收到的值符合 params 的类型。

The params determine which values your API can accept. Interestingly, Milkio will automatically use Typia to help ensure that the received values match the types specified in the params.

因此，你只需要编写纯粹的 TypeScript 类型就好了，泛型和任何 TypeScript 高级类型你都可以随意使用。

So, you just need to write pure TypeScript types, and you can freely use generics and any advanced TypeScript types.

ts
action(
  params: { id: string; gender: string; email: string; age: number; },
  context
) {
ts
action(
  params: { id: string; gender: string; email: string; age: number; },
  context
) {

Typia 最引人注目的地方是拥有 Type Tags，你可以用它进一步增强你的 params 类型。

One of the most notable features of Typia is its use of Type Tags, which can be used to further enhance your params types.

ts
action(
  params: {
    id: string & typia.tags.Format<"uuid">;
    gender: "male" | "female" | "other";
    email: string & typia.tags.Format<"email">;
    age: number & typia.tags.ExclusiveMinimum<19> & typia.tags.Maximum<100>;
  },
  context
) {
ts

    email: string & typia.tags.Format<"email">;
    age: number & typia.tags.ExclusiveMinimum<19> & typia.tags.Maximum<100>;
  },
  context
) {

像这样，限制字符串长度、是否是邮件…… 都不在话下。

Restricting string length and checking if it is an email are all a piece of cake!

你可能会好奇，TypeScript 的类型会在编译为 JavaScript 时全部剔除，真正所运行的代码，是没有任何类型的。Typia 是怎样实现这样的"魔法"的？

You might be curious about how TypeScript's types are completely removed during compilation to JavaScript, and the actual running code has no types at all. How does Typia achieve this kind of "magic"?

Typia 会分析你的代码，并执行 AOT 编译。将验证部分的代码转换为专有的适合你类型的代码。这会让编写 params 更加容易，且运行更快。你可以阅读 Typia 的 AOT Compilation 章节，来更详细地了解。

Typia will analyze your code and perform AOT compilation. It will transform the verification part of the code into proprietary code that is tailored to your types. This will make writing params easier and improve runtime performance. You can read the AOT Compilation section in Typia's documentation for more detailed information.

Typia 的 AOT 编译依赖于 TSC，这确实会影响我们对运行时和打包工具的选择。但由于 Milkio 已经在 生成阶段 中内置了对 Typia 的支持，因此，你可以自由地选择用任何你喜欢的运行时和打包工具，而非一定使用 TSC。

Typia's AOT compilation relies on TSC, which does indeed impact our choice of runtime and packaging tools. However, since Milkio has already built-in support for Typia in the generation phase, you are free to choose any runtime and packaging tools you prefer, rather than being limited to TSC.

Context ​

Context ​

Context 是一个对象，它包含了与当前请求有关的一些信息和一些有用的方法。在不同的请求之间，Context 都是不同的。

Context is an object that contains information and useful methods related to the current request. The Context object is different for each request, meaning it can vary across different requests.

ts
action(
  params: { ... },
  context
) {
ts
action(
  params: { ... },
  context
) {

例如，我们最常用的就是 context.logger，它类似于 console.log，想要为当前请求写入一条日志，只要这样：

For example, the most commonly used one is context.logger, which is similar to console.log. If you want to write a log for the current request, you just need to do this:

ts
context.logger.log('Hello, Milkio!')
ts
context.logger.log('Hello, Milkio!')

logger 的用法几乎和 console 一样，可以阅读 Logger 来了解它。

The usage of the logger is almost the same as the console. You can read Logger to learn about it.

扩展 Context ​

Extend Context ​

你还可以扩展 Context，你完全可以把它当作是一个在请求间共享的"globalThis"，例如，在 中间件 中读取用户的数据，并将其附加到 context.user 中。在 API 中，你可以直接通过 context.user 来获取当前用户的信息。

You can also extend Context and treat it as a "globalThis" shared between requests. For example, you can read user data in middleware and attach it to context.user. In your API, you can then directly access the current user's information through context.user.

不要忘了编辑 /src/context.ts 文件来补充 Context 的类型，但是，请注意，扩展 Context 的类型并不意味着数据实际上存在于 Context 中。除了 Context 的默认属性之外，你需要自己编写代码将数据附加到 Context 上。

Don't forget to edit the /src/context.ts file to supplement the type of Context. However, please note that extending the type of Context does not mean that the data actually exists in Context. Besides the default properties of Context, you need to write code yourself to attach data to Context.

ts
// 文件: /src/context.ts
import type { MilkioContext } from "milkio";

export type Context = {
  // 添加了用户信息类型
  user: { 
    id: string; 
    name: string; 
    email: string; 
  }; 
  // ...
} & MilkioContext;
ts
// file: /src/context.ts
import type { MilkioContext } from "milkio";

export type Context = {
  // Add user information type
  user: { 
    id: string; 
    name: string; 
    email: string; 
  }; 
  // ...
} & MilkioContext;
