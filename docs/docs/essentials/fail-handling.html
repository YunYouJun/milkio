Failure Handling ​

Failure Handling ​

在我们编写后端应用的过程中，通常会遇到两种失败情况。一种则是与业务相关的失败，比如用户不存在或订单已支付。另一种是由于意料之外错误引起的失败，例如网络问题或堆栈溢出。

In the process of writing backend applications, we usually encounter two types of failure scenarios. One is failures related to business logic, such as user not found or order already paid. The other is failures caused by unexpected errors, such as network issues or stack overflow.

对于业务相关的失败情况，我们可以使用 reject 方法来主动抛出错误，以向客户端提供明确的失败响应。

For business-related failures, we can use the reject method to actively throw an error, providing a clear failure response to the client.

ts
if (!user) throw reject("NOT_FOUND", undefined);
ts
if (!user) throw reject("NOT_FOUND", undefined);

通过这种方式，我们能够更精准地告知客户端请求失败的原因，从而使系统更加健壮和可靠。对于调用者，将得到这样的响应：

By using this approach, we can inform the client more accurately about the reasons for the request failure, making the system more robust and reliable. The caller will receive a response like this:

json
{
  "success": false,
  "fail": {
    "code": "NOT_FOUND",
    "message": "Not Found",
    "data": undefined
  }
}
json
{
  "success": false,
  "fail": {
    "code": "NOT_FOUND",
    "message": "Not Found",
    "data": undefined
  }
}

这样，调用者可以通过 code.success 简单地了解本次请求是否成功，并通过 fail.code 判断失败的具体类型。这种设计方式有助于提供清晰的请求状态信息，以便调用者能够更好地处理和响应失败情况。

In this way, the caller can easily understand whether the request was successful through code.success, and determine the specific type of failure through fail.code. This design approach helps provide clear request status information, enabling the caller to better handle and respond to failure situations.

失败码 ​

Fail Code ​

在 /src/fail-code.ts 中，我们定义了所有可选的失败码。

In /src/fail-code.ts, we define all the optional failure codes.

其中，每个键都代表一个特定的失败码，对应的值是一个方法。这个方法需要返回一个字符串，用于说明失败的具体原因。通过这种设计，我们可以清晰地组织和管理不同的失败码，并为每个失败码提供详细的描述功能。

Among them, each key represents a specific failure code, and the corresponding value is a method. This method needs to return a string that explains the specific reason for the failure. With this design, we can organize and manage different failure codes clearly and provide detailed descriptions for each failure code.

ts
// 文件: /src/fail-code.ts

import type { MilkioFailCode } from "milkio";

export const failCode = {
  "NETWORK_ERROR": () => "Network Error",
  "INTERNAL_SERVER_ERROR": () => "Internal Server Error",
  "NOT_FOUND": () => "Not Found",
  "NOT_ALLOW_METHOD": () => "Not Allow Method",
  "TYPE_SAFE_ERROR": (p: { path: string; expected: string; value: string }) => `Parameter Error: The current value is '${p.value}', which does not meet '${p.expected}' requirements`,
  "BUSINESS_FAIL": (message: string) => `${message}`,
  // 你可以在这里添加自己的失败码
  // ...
} satisfies MilkioFailCode;

参数 ​

Parameters ​

在调用 reject 方法时，你可以通过传入的参数来改变最终向客户端响应的消息信息。这样，你可以根据需要灵活地定制错误消息。

In the reject method, you can pass in the parameters to change the message sent to the client at the end. This way, you can customize the error message according to your needs.

ts
throw reject("ACCESS_EXCEEDED", { path: "/foo/bar", counter: 64 });
ts
throw reject("ACCESS_EXCEEDED", { path: "/foo/bar", counter: 64 });

需要注意的是，该方法只允许传递一个参数。如果你有多个参数需要返回，请尝试像上面的例子一样，通过返回一个对象来实现。这样，你可以将多个参数封装在一个对象中，以便进行传递和处理。

Please note that this method only allows passing one parameter. If you have multiple parameters to return, you can try to achieve it by returning an object as shown in the example above. This way, you can encapsulate multiple parameters in one object for passing and handling.

ts
// 文件: /src/fail-code.ts
export const failCode = {
  "ACCESS_EXCEEDED": (p: { path: string; counter: number }) =>
    `You are visiting too frequently. This path "${p.path}" is only allowed to be accessed ${p.counter} times.`,
} satisfies MilkioFailCode;
ts

Meanwhile, these parameters will also be returned to the client so that the client can retrieve and further process them. In the end, the client will receive a response similar to the following, including the concatenated message and data:

意料之外的错误 ​

Unexpected error ​

对于任何未被捕获的错误，尽管，你可以在日志中看到详细的错误原因和错误栈，但客户端将不会得到真实的错误原因。客户端所获得的将是固定的 INTERNAL_SERVER_ERROR 失败码。这可以有效地避免泄露敏感信息。

For any uncaught errors, although you can see detailed error reasons and error stacks in the logs, the client will not receive the actual error reason. The client will receive a fixed INTERNAL_SERVER_ERROR failure code. This effectively avoids leaking sensitive information.
