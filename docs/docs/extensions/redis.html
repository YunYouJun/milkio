Redis ​

Redis ​

Redis 可以临时缓存某些数据，并在适当的时候自动失效。巧妙地运用能够显著提升你的应用性能。

Redis can temporarily cache certain data and automatically expire it at the appropriate time. Cleverly using it can significantly improve the performance of your application.

入门 ​

Get Started ​

我们需要安装 Milkio 与 Redis 所结合的包：

We need to install the package that combines Milkio with Redis:

sh
bun i milkio-redis
sh
bun i milkio-redis

在你的 /src/uses 目录中，创建一个 drizzle.ts，并将下面的代码粘贴进去。其中的 url 不要忘记修改哦。

In your /src/uses directory, create a drizzle.ts file and paste the following code into it. Don't forget to modify the url.

ts
import { defineRedisClient } from "milkio-redis"
import { configRedis } from "../config/redis"

export const redisClient = defineRedisClient({
  url: "redis://127.0.0.1:6379",
  initTimeout: 3000,
})
ts
import { defineRedisClient } from "milkio-redis"
import { configRedis } from "../config/redis"

export const redisClient = defineRedisClient({
  url: "redis://127.0.0.1:6379",
  initTimeout: 3000,
})

获取原生 Redis 客户端 ​

Get Native Redis Client ​

我们可以直接使用原生的 Redis 客户端 来操作 Redis，你可以使用所有的 Redis 命令。像这样：

We can directly use the native Redis client (https://github.com/redis/node-redis) to operate Redis. You can use all Redis commands. Like this:

ts
(await redisClient).HSET('key', 'field', 'value')
ts
(await redisClient).HSET('key', 'field', 'value')

也可以使用经过 Milkio 精心封装后的缓存功能，拥有完整的类型支持，代码写起来更加高效简单，并且无需关注所存储的值的类型。Milkio 会自动帮你将值使用 TSON 序列化，并在取出时反序列化。

You can also use the caching functionality provided by Milkio, which is carefully encapsulated and supports various types. It makes the code more efficient and simple, without the need to worry about the type of the stored values. Milkio will automatically serialize the values using TSON and deserialize them when retrieved.

缓存 ​

Cache ​

你可以使用 defineCache 方法来定义一个缓存。比如，你可以缓存一个布尔值，用于表示是否开启了维护模式。

You can use the defineCache method to define a cache. For example, you can cache a boolean value to indicate whether the maintenance mode is enabled.

ts
import { defineCache } from "milkio-redis";

const cache = defineCache<boolean>("enable-maintenance-mode");

await cache.set(true, 100);
console.log(await cache.get());
// echo: true
ts
import { defineCache } from "milkio-redis";

const cache = defineCache<boolean>("enable-maintenance-mode");

await cache.set(true, 100);
console.log(await cache.get());
// 输出: true

命名空间缓存 ​

Namespace Cache ​

有时候，我们需要存储集合数据的场景，比如为每个用户的个人信息创建缓存。这样可以方便地以用户为单位进行数据缓存，提高读取效率。

In some scenarios, we need to store collection data. For example, we can create a cache for each user's personal information. This can help us to cache data in a user-specific manner.

ts
import { defineNamespaceCache } from "milkio-redis";

const cache = defineNamespaceCache<UserInfo>("user-info");
type UserInfo = {
  age: number;
  email: string;
};

await cache.set("alice", { age: 18, email: "alice@gmail.com" }, 100);
console.log(await cache.get("alice"));
// 输出: { age: 18, email: 'alice@gmail.com' }

await cache.del();
console.log(await cache.get());
// 输出: undefined
ts
import { defineNamespaceCache } from "milkio-redis";

const cache = defineNamespaceCache<UserInfo>("user-info");
type UserInfo = {
  age: number;
  email: string;
};

await cache.set("alice", { age: 18, email: "alice@gmail.com" }, 100);
console.log(await cache.get("alice"));
// 输出: { age: 18, email: 'alice@gmail.com' }

await cache.del();
console.log(await cache.get());
// 输出: undefined

结果缓存 ​

Result Cache ​

结果缓存可以缓存函数执行的结果。例如，我们可以缓存 SQL 查询的结果，或者缓存一些耗时的计算，以减少重复工作并加快响应速度。下面的代码中，我们传递了 4 个参数，分别是键名、缓存过期时间、缓存的实际过期时间以及函数作为参数。在缓存未过期之前，函数将不会被重复执行，而是从缓存中获取结果。

The result cache can cache the execution result of a function. For example, we can cache the results of SQL queries or cache some time-consuming calculations to reduce redundant work and improve response speed. In the following code, we pass four parameters: the key name, cache expiration time, actual expiration time of the cache, and the function as a parameter. Before the cache expires, the function will not be executed repeatedly, but rather the result will be retrieved from the cache.

ts
const result = defineResultCache("your-key", 60, 60, async () => {
  const now = new Date().getTime();
  await new Promise((resolve) => setTimeout(resolve, 1000)); // 等待 1 秒
  return now;
});
ts
const result = defineResultCache("your-key", 60, 60, async () => {
  const now = new Date().getTime();
  await new Promise((resolve) => setTimeout(resolve, 1000)); // sleep for 1 second
  return now;
});

当执行 defineResultCache 代码时，若缓存未过期，则从缓存中读取结果。如果缓存过期，则会运行方法获取结果，并重新缓存起来。

When executing the defineResultCache code, if the cache is not expired, the result will be read from the cache. If the cache is expired, the method will be executed to obtain the result and then re-cached.

为了减少缓存雪崩的概率，你需要设置两个过期时间：TTL 和 realTTL。

To reduce the likelihood of cache avalanche, you need to set two expiration times: TTL and realTTL.

当 TTL 未过期时，进入正常缓存阶段，所有结果都来自缓存。

When the TTL (Time To Live) has not expired, it enters the normal caching stage, and all results are obtained from the cache.

当 TTL 过期后，进入缓存过期阶段，此时 Milkio 允许执行一个请求重新计算结果。在重新计算阶段期间，其他请求仍获取旧的缓存数据，直到重新计算阶段结束并更新缓存，回到正常缓存阶段。

After the TTL expires, entering the cache expiration stage, Milkio allows for executing a request to recalculate the result. During the recalculation phase, other requests still retrieve old cached data until the recalculation phase is completed and the cache is updated, returning to the normal caching phase.

重新计算有可能失败，导致回到缓存过期阶段。下一个到达的请求将重新进入重新计算阶段。默认情况下，重新计算阶段持续 6 秒。若超过此时间，则认为重新计算失败。可以通过 options.realGetInterval 修改超时时间。

Recomputing is possible to fail, resulting in returning to the cache expiration stage. The next incoming request will reenter the recomputing stage. By default, the recomputing stage lasts for 6 seconds. If it exceeds this time, it is considered as recomputing failure. The timeout can be modified by options.realGetInterval.

若在相当一段时间内没有请求获取结果，或持续获取失败，直到达到 TTL + realTTL 的过期时间后，缓存结果将被彻底删除。

If there is no request to retrieve the result for a considerable period of time, or if the retrieval fails continuously until the expiration time of TTL + realTTL is reached, the cached result will be completely deleted.
