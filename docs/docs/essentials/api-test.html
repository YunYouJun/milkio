API Test ​

API Test ​

嗨！你讨厌编写测试吗？我也非常讨厌编写测试，在创造 Milkio 之前。因为我没有接触过任何一个框架的设计是重视测试的，这迫使我必须编写大量的 mock 和许多重复的代码，或者必须做一些无意义的工作，才能使测试通过。

Hi! Do you hate writing tests? I used to hate writing tests too, before creating Milkio. Because I haven't been exposed to any framework that prioritizes testing, it forces me to write a lot of mocks and repetitive code, or I have to do some meaningless work just to make the tests pass.

所以，我在设计 Milkio 时，把测试做的尽可能地易用和简单。你可以一边开发你的 API 一边使用测试功能来调试你的 API，当你调试完成一个功能点后，就可以把它保留为测试用例。你可能不再需要 Postman 或者其他类似的接口调试工具了！

So, when designing Milkio, I made the testing feature as user-friendly and simple as possible. You can develop your API while using the testing functionality to debug it. Once you have debugged a specific feature, you can save it as a test case. You may no longer need Postman or other similar interface debugging tools!

或许，你已经发现了……？在 /src/hello-world/say.ts 的 api 的下面，还有一个 test 存在。是的，每一个 API 都可以编写一个测试，尽管这不是必需的，但是却是我们所推荐的最佳实践。

Perhaps you have already noticed...? Below the API in /src/hello-world/say.ts, there is also a test. Yes, you can write a test for each API, although it is not required, it is recommended as a best practice.

INFO

你可以通过点击 VS Code 右上角的 烧瓶图案 来运行这个 API 的测试。烧瓶图案只有在 API 拥有至少一个测试时才会显示。你也可以通过快捷键 (⌘ + ⇧ + R) 或 (Alt + R) 来运行测试。

INFO

You can run the tests for this API by clicking on the flask icon in the top right corner of VS Code. The flask icon will only be displayed if the API has at least one test. You can also run the tests by using the shortcut (⌘ + ⇧ + R) or (Alt + R).

ts
export const api = defineApi({
  meta: {},
  async action(params: { by: string }, context) {
    return `Hello ${params.by}!`
  }
})

export const test = defineApiTest(api, [{
  name: "你的测试名称",
  handler: async (test) => {
    // 你的测试代码..
  }
}])
ts
export const api = defineApi({
  meta: {},
  async action(params: { by: string }, context) {
    return `Hello ${params.by}!`
  }
})

export const test = defineApiTest(api, [{
  name: "Your test name",
  handler: async (test) => {
    // Your test code..
  }
}])

每个 API 都可以编写多个测试，因为 defineApiTest 的第二个参数是一个数组。当测试运行时，它们不会是并行的，因为 API 测试和单元测试不同，API 测试依赖于数据库等很难充分 mock 的对象，因此，我们需要通过清空数据库再运行第二个测试等方式进行，来确保测试之间不会互相影响。

Each API can have multiple tests written for it because the second parameter of defineApiTest is an array. When the tests are run, they are not parallel because API tests, unlike unit tests, depend on objects such as databases that are difficult to fully mock. Therefore, we need to ensure that the tests do not interfere with each other by clearing the database before running the second test, among other methods.

测试 API ​

Test API ​

在测试中，你可以很方便地执行你在上面编写的 API，你甚至连 API 的路径都不需要编写。同时，可以通过 test.reject 来使测试失败。

In testing, you can conveniently execute the APIs you have written above without even needing to write the API paths. Additionally, you can use test.reject to make the test fail.

ts
handler: async (test) => {
  const result = await test.execute({ by: "Milkio" })
  if (!result.success) return test.reject(`这里是你的 API 测试失败的原因`)
}
ts
handler: async (test) => {
  const result = await test.execute({ by: "Milkio" })
  if (!result.success) return test.reject(`The reason for your API test failure here is`)
}

甚至，你也不需要为你的接口编写测试的参数，可以通过 test.randParams 来随机生成！如果你认为生成的参数不合适，你可以通过 Typia Type Tags 来收紧你的类型限制，这样就会生成出更符合你需要的参数。

Even better, you don't even need to write test parameters for your interface. You can use test.randParams to generate random ones! If you find the generated parameters unsuitable, you can tighten your type restrictions using Typia Type Tags, which will generate parameters that better suit your needs.

ts
handler: async (test) => {
  const result = await test.execute(await test.randParams())
  if (!result.success) return test.reject(`这里是你的 API 测试失败的原因`)
}
ts
handler: async (test) => {
  const result = await test.execute(await test.randParams())
  if (!result.success) return test.reject(`The reason for your API test failure here is`)
}

测试其他 API ​

Test other APIs ​

你或许需要在测试过程中，执行其他 API。例如，你在测试的是一个更新 API，你需要在此之前先创建数据，才能进行更新。只需要使用 executeOther 方法，就可以轻易地执行其他 API。也可以使用 randOtherParams 来为它生成随机的参数：

You may need to execute other APIs during the testing process. For example, if you are testing an update API, you need to create data before you can perform the update. You can easily execute other APIs using the executeOther method. You can also use randOtherParams to generate random parameters for it.

ts
handler: async (test) => {
  const params = await test.randOtherParams('user/create')
  const createResult = test.executeOther('user/create', params)
  if (!createResult.success) return test.reject(`创建过程中失败了`)
  params.name = 'bar' // 修改参数
  const updateResult = await test.execute(params)
  if (!updateResult.success) return test.reject(`更新过程中失败了`)
}
ts
handler: async (test) => {
  const params = await test.randOtherParams('user/create')
  const createResult = test.executeOther('user/create', params)
  if (!createResult.success) return test.reject(`Failed during the creation process`)
  params.name = 'bar' // Modify the parameter
  const updateResult = await test.execute(params)
  if (!updateResult.success) return test.reject(`Failed during the update process`)
}

运行所有测试 ​

Run All Tests ​

你可以在 VS Code 中，通过点击左下角的 ▶️ Milkio 按钮，选择 Test All APIs 来一次性运行所有的 API 测试。

You can run all API tests at once in VS Code by clicking on the ▶️ Milkio button at the bottom left corner and selecting Test All APIs.

INFO

你也可以通过快捷键 (⌘ + ⇧ + L) 或 (Alt + L) 唤出此菜单。

生命周期挂钩 ​

Lifecycle hooks ​

在你的 /src/api-test.ts 文件中，你可以在测试启动时，或者在每个测试运行之前，执行代码。我们通常会在每个测试运行之前，清空数据库来保证下一次测试时数据库是干净的。

In your /src/api-test.ts file, you can execute code during test startup or before each test run. We usually clear the database before each test run to ensure that the database is clean for the next test.

ts
// 文件: /src/api-test.ts
export default {
  async onBootstrap() {
    // ..
  },
  async onBefore() {
    return {
      // 这里返回的内容将被混合到测试对象中
    }
  }
}
ts
// file: /src/api-test.ts
export default {
  async onBootstrap() {
    // ..
  },
  async onBefore() {
    return {
      // The content returned here will be mixed into the test object
    }
  }
}

其中，对于 onBefore，你可以返回一个对象，这个对象将被混入到到 test 对象当中。你可以在这里添加测试时可用的工具。

For onBefore, you can return an object that will be mixed into the test object. You can add any tools that are available for testing here.
